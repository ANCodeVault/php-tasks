
### Структурные шаблоны проектирования (Structural)

**Реестр (Registry)**

**Registry (реестр)** — это дизайн-паттерн, который используется в программной разработке для централизованного управления и доступа к общим объектам или экземплярам. Он позволяет создать единое хранилище, где объекты регистрируются с уникальными идентификаторами, а другие части программы могут получать к ним доступ через этот реестр.

**Назначение**

Для реализации централизованного хранения объектов, часто используемых во всем приложении, как правило, реализуется с помощью абстрактного класса только c статическими методами (или с помощью шаблона Singleton). Помните, что это вводит глобальное состояние, которого следует избегать. Используйте Dependency Injection вместо Registry.

**Основные компоненты паттерна**

* Registry — центральное хранилище, где объекты регистрируются и хранятся. Может быть реализовано как класс, поддерживающий коллекцию (например, словарь или карту) объектов, индексированных уникальными ключами.
* Ключи/идентификаторы — уникальные идентификаторы, используемые для регистрации и получения объектов из реестра. Могут быть строками, перечислениями или другими типами, обеспечивающими уникальность в рамках реестра.
* Зарегистрированные объекты — сами экземпляры или объекты, которые хранятся в реестре. Они могут быть разных типов в зависимости от потребностей приложения.
* Интерфейс регистрации — методы, предоставляемые реестром для добавления объектов (например, register, add, put).
* Интерфейс поиска — методы реестра для получения объектов.
* Инициализация и настройка — процесс настройки реестра, обычно на этапе запуска приложения. Может включать загрузку конфигураций, инициализацию объектов по умолчанию, предварительную регистрацию определённых ключевых объектов.

**Принцип работы**

* Инициализация реестра — в начале работы приложения или при необходимости создаётся объект реестра, который служит централизованным хранилищем.
* Регистрация объектов — объекты, которые должны использоваться в разных частях приложения, регистрируются в реестре. Каждому объекту присваивается уникальный идентификатор (ключ).
* Централизованный доступ — другие компоненты или модули приложения могут получать зарегистрированные объекты через реестр, используя их ключи (например, get(key)).

**Преимущества паттерна**

* Упрощение управления объектами за счёт предоставления единого места для регистрации и получения экземпляров.
* Снижение зависимостей между частями системы.
* Возможность повторного использования объектов вместо создания новых при каждом обращении.
* Упрощение добавления новых типов объектов в систему без изменения существующего кода.
* Управление жизненным циклом объектов (например, управление синглтонами).

**Некоторые нюансы**

* Глобальное состояние — Registry вводит глобальное состояние, что может быть нежелательно в некоторых случаях. В таких ситуациях иногда рекомендуется использовать альтернативные подходы, например, Dependency Injection.
* Поточная безопасность — если реестр доступен из нескольких потоков, необходимо обеспечить его потокобезопасность.
* Управление жизненным циклом объектов — реестр может решать, когда объекты должны создаваться и уничтожаться.
